# 第5章 Note

## 全体所感

「多様な環境で成功を収める」章。**第一部で最も気づきが多く、整理になる章**。プロダクト/プラットフォーム、戦時/平時の比較は、無意識に扱っていることの言語化として非常に有用。じっくり原文を読むことを推奨。

## プロダクトチーム vs プラットフォームチーム

この対比はプラットフォームエンジニアとして重要な視点を提供する。

### プロダクトチーム
- 顧客に直接価値を届ける
- ビジネスインパクトを示しやすい（プロフィットセンター寄り）
- 機能リリースが成果の指標になりやすい

### プラットフォームチーム
- 開発者チームを「顧客」として支援
- 間接的な価値提供（コストセンター寄り）
- 価値の可視化が課題（開発者生産性、信頼性、セキュリティなど）

両者の違いを理解することで、自分の立場に応じた成果の示し方が明確になる。

## 戦時 vs 平時

組織の状態によって求められる行動が異なるという整理。

### 平時
- 長期的な投資、技術負債の返済
- プロセスの整備、ドキュメント化
- 人材育成、組織づくり

### 戦時
- 短期的な成果、緊急対応
- プロセスの簡略化、スピード重視
- 既存リソースの最大活用

プラットフォームチームは「平時」にこそ投資が認められやすいが、「戦時」には存在意義が問われやすい（コストセンターとして削減対象になりうる）。この動学を理解しておくことで、組織の状態に応じた振る舞いができる。

## プロダクト志向のエンジニア

プラットフォームチームであっても「プロダクト志向」の視点は重要。

- 開発者チームのニーズを理解する
- ツールやプロセスを「プロダクト」として提供する
- フィードバックを受けて改善するサイクルを回す

第21章（ビジネスを理解する）の内容とも通じる視点。
