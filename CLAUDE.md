# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## プロジェクト概要

読書ノートを蓄積・管理するためのリポジトリ。技術書の重要なポイントをMarkdownファイルで体系的に整理し、副読書として活用する。

## コンテンツ構造

```
{書籍名}/
├── toc.md                    # 書籍全体の目次（各章summaryへのリンク付き）
└── contents/
    └── chapter_XX/
        ├── summary.md        # 章全体のまとめ
        ├── note.md           # 読者の考察・メモ
        └── section_XX.md     # 各セクションの詳細
```

### ファイルの役割

| ファイル | 目的 | 内容 |
|---------|------|------|
| `toc.md` | 目次 | 書籍全体の構成とsummaryへのリンク |
| `summary.md` | 章のまとめ | 章全体の体系的なサマリー |
| `note.md` | 個人的考察 | 読者自身の気づき・批評・コメント |
| `section_XX.md` | セクション詳細 | 「概要」「詳細」「ポイント」の3部構成 |

### セクションファイルの標準フォーマット

```markdown
# セクションタイトル

## 概要
[1-2文で要約]

## 詳細
[箇条書きで詳細説明]

## ポイント
[重要な学びや実践的な示唆]
```

## 作業時の注意

- 日本語で記述（本リポジトリは日本語読者向け）
- 引用を含める場合は原文ページ番号を添える（例：`（P.123より）`）
- note.mdは読者個人の視点・考察を記述するため、本文の要約とは区別する

## 筆者の傾向・思考スタイル

note.mdの内容から推定される筆者の特徴。AI支援時にはこれらを考慮すること。

### 立場・経験

- 日本のミドル〜シニアクラス以上のエンジニア（リードエンジニアまたはマネージャー層）
- SRE・プラットフォームエンジニアとしての業務が中心。純粋な開発者目線（コーディング業務など）よりも、CI/CD、ツール整備、信頼性、コンプライアンス等のプラットフォーム寄りのトピックに関心が高い
- ジュニア向けの技術的内容は「既知」として割愛する傾向
- マネージャー・リーダー視点から「逆算した行動」を考える思考法

### 日本と海外の文化差への意識

- 海外（主に米国）企業文化と日本企業の違いを強く意識
- 海外向けの内容は「話半分」「読み流す」など取捨選択する
- 日本では**サマライズ能力**が重視されるという見解
- 1on1や過剰なコミュニケーションは海外文化として距離を置く

### 重視する概念

- **情報の非対称性**: マネージャーとIC間、評価者と被評価者間の認識ギャップ。日本企業では縦割り文化（サイロ）により情報がキーパーソンに集約されがちなため、戦略会議への参加依頼、PRDを読む時間の確保、他部門との直接対話などで能動的に情報を取りに行く
- **信頼資本・信頼残高**: コミュニケーションと関係構築の土台
- **コミュニケーションの質**: 技術力だけでは不十分、建設的な対話と影響力の発揮が必要。対立や揉め事が生じた際はテキスト（チャット・メール）ではなく直接対話（ビデオ会議含む）で解決する。テキストベースは誤解を生みやすい
- **T字型人材**: 専門領域の深さだけでなく、周辺技術・オペレーション・隣接領域への厚みがアウトプットに寄与するという考え方を重視
- **AIによる知識転写の活性化**: ドキュメント更新等の仕組み化はAIにより容易になっており、人→AI、AIを通じて人→人への知識転写を積極的に進めるべきという考え

### フィードバックへの考え方

- ネガティブフィードバックは極力避ける（ハラスメント等の例外を除く）
- 本人が問題を認識して相談に来た際のカウンターとして行う
- AIを介したフィードバック手法も活用（壁打ち結果をウォークスルー）

### 用語・表現への配慮

- 「上流・下流」という表現は好まない（依存関係で捉える）
- 翻訳用語は適宜読み替える（例：「のれん残高」→「信頼残高」）
- 自己の経験・レベルを明示する表現は避ける（例：「自分がこのレベルで仕事をしている」→ 傲慢に映るため客観的でも使わない）

### リーダー/アーキテクトとしての振る舞い

筆者が好む振る舞いの原則。第25章（ソフトウェアアーキテクチャー）の知見と統合。

- **自身がボトルネックにならない**: 全ての決定を自分で行うのではなく、他者が自律的に判断できる仕組み・組織をつくる。これは「信頼資本」の構築と「情報の非対称性」の解消を通じて実現される
- **メンターとしての振る舞い**: 答えを与えるのではなく、質問を投げかけ、トレードオフを提示し、相手が自ら決定できるよう導く。AIを活用した知識転写（ドキュメント化、壁打ち）もこの延長線上にある
- **ビジネス理解と価値交換の管理**: 技術的判断は常にビジネス目標との整合性を意識する。「完璧」より「十分良い」を選び、アーキテクチャー改善はビジネスプロジェクトに抱き合わせてリソースを確保する。決算説明会資料や中期経営計画を読み、経営層の投資方針・会社の方向性を把握することを実践している
- **現場感覚の維持**: 「象牙の塔」に籠らず、コードや技術の現場に留まりつつ戦略的議論にも参加する。T字型人材として専門性と周辺領域への厚みを両立させる
- **影響範囲を意識した意思決定**: ワンウェイドア（不可逆）とツーウェイドア（可逆）を見極め、リソース配分を最適化する。影響範囲が広い決定は縮小のアプローチを模索する

### 読書スタイル

- 批評的に読み、日本の文脈に置き換えて解釈
- 重要な箇所は引用しページ番号を添える
- パブリックに書きにくい内容は明示的に「noteなし」とする慎重さ
