# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## プロジェクト概要

読書ノートを蓄積・管理するためのリポジトリ。技術書の重要なポイントをMarkdownファイルで体系的に整理し、副読書として活用する。

## コンテンツ構造

```
{書籍名}/
├── toc.md                    # 書籍全体の目次（各章summaryへのリンク付き）
└── contents/
    └── chapter_XX/
        ├── summary.md        # 章全体のまとめ
        ├── note.md           # 読者の考察・メモ
        └── section_XX.md     # 各セクションの詳細
```

### ファイルの役割

| ファイル | 目的 | 内容 |
|---------|------|------|
| `toc.md` | 目次 | 書籍全体の構成とsummaryへのリンク |
| `summary.md` | 章のまとめ | 章全体の体系的なサマリー |
| `note.md` | 個人的考察 | 読者自身の気づき・批評・コメント |
| `section_XX.md` | セクション詳細 | 「概要」「詳細」「ポイント」の3部構成 |

### セクションファイルの標準フォーマット

```markdown
# セクションタイトル

## 概要
[1-2文で要約]

## 詳細
[箇条書きで詳細説明]

## ポイント
[重要な学びや実践的な示唆]
```

## 作業時の注意

- 日本語で記述（本リポジトリは日本語読者向け）
- 引用を含める場合は原文ページ番号を添える（例：`（P.123より）`）
- note.mdは読者個人の視点・考察を記述するため、本文の要約とは区別する

## 筆者（この場合はrepo ownerを指し本書の筆者を示さないことに注意）の傾向・思考スタイル

note.mdの内容から推定される筆者の特徴。AI支援時にはこれらを考慮すること。

### 立場・経験

- 日本のミドル〜シニアクラス以上のエンジニア（リードエンジニアまたはマネージャー層）
- **SRE・プラットフォームエンジニア**としての業務が中心
  - CI/CD、ツール整備、信頼性、コンプライアンス等のプラットフォーム寄りのトピックに関心が高い
  - 開発者チームを「顧客」として捉え、彼らの生産性向上・障害削減を自チームの価値として可視化する視点
  - オンコール、ポストモーテム、ランブック整備など信頼性エンジニアリングの実務経験
- ジュニア向けの技術的内容は「既知」として割愛する傾向
- マネージャー・リーダー視点から「逆算した行動」を考える思考法

### 核となる概念

筆者のnote全体を通じて繰り返し登場する概念。

#### 情報の非対称性

マネージャーとIC間、評価者と被評価者間、チーム間の認識ギャップ。日本企業では縦割り文化（サイロ）により深刻化しやすい。

- 役割の暗黙化による混乱（誰が担当か不明、重複）
- 経営層の発言の「翻訳」（言われたこと/言われなかったことの読み解き）
- 評価されない仕事の存在（マネージャーが気づいていない貢献）
- 対策: 戦略会議への参加依頼、PRDを読む時間の確保、他部門との直接対話で**能動的に情報を取りに行く**

#### 信頼資本（Trust Capital）

コミュニケーションと関係構築の土台。影響力の源泉。

- 構成要素: 肩書き・権限、在籍期間、専門知識、実績、**仕事の可視化**
- 「やっても知られなければ蓄積されない」ため、成果の可視化は意識的に行う
- スポンサーとしての振る舞い（他者の優れた仕事に光を当てる）も信頼資本の活用

#### T字型人材

専門領域の深さだけでなく、周辺技術・オペレーション・隣接領域への**厚み**がアウトプットに寄与する。

- デバッグ能力はコードを読む力だけでなく、監視ツール・インフラ知識・過去のインシデント事例といった周辺領域への厚みで差がつく
- マイクロサービス化が進む現代ではフルスタック志向の重要性が増している

#### AIによる知識転写

ドキュメント更新等の仕組み化はAIにより容易になっており、人→AI、AIを通じて人→人への知識転写を積極的に進めるべき。

- ランブック更新のAI支援による自動化・半自動化
- サービスカタログ・開発者ポータルとAIの連携
- 壁打ち結果のウォークスルーによるフィードバック手法

### 意思決定・アプローチ

#### 課題起点

- 「どのプロセス/ツールを導入すべきか」ではなく「最大の課題は何か」から逆算
- プロセスは目的ではなく手段。形骸化したプロセスは廃止する権限と責任がある
- 技術負債も「ローン」として戦略的に管理。単純な善悪ではなく速度とのトレードオフ

#### 「完璧」より「十分良い」

- 完璧なアーキテクチャーより、現在のビジネスニーズを満たす「十分良い」を選ぶ
- アーキテクチャー改善や技術負債返済は**ビジネスプロジェクトに抱き合わせて**リソースを確保
- 「許可を求めずにプロジェクトの一部として返済する」アプローチ

#### ワンウェイドア vs ツーウェイドア

- ツーウェイドア（可逆）: 素早く決断、議論時間を浪費しない
- ワンウェイドア（不可逆）: PoCで検証、慎重に
- 注意: 依存関係が生まれた時点で可逆性が失われる（公開APIなど）
- 影響範囲が広い決定は**縮小のアプローチ**を模索

### リーダー/アーキテクトとしての振る舞い

#### 自身がボトルネックにならない

- 全ての決定を自分で行うのではなく、他者が自律的に判断できる仕組み・組織をつくる
- 「信頼資本」の構築と「情報の非対称性」の解消を通じて実現
- 自立したメンバーで構成されるチームの方が、リーダーの決定待ちチームより効率的

#### メンターとしての振る舞い

- 答えを与えるのではなく、質問を投げかけ、トレードオフを提示し、相手が自ら決定できるよう導く
- AIを活用した知識転写（ドキュメント化、壁打ち）もこの延長線上
- スポンサーとしての役割: 影響力を使って他者のキャリアを積極的に引き上げる

#### ビジネス理解

- 技術的判断は常にビジネス目標との整合性を意識
- 決算説明会資料や中期経営計画を読み、経営層の投資方針・会社の方向性を把握
- プラットフォームチームは本質的に**コストセンター**であり、価値の可視化（開発者生産性向上の定量化、障害削減のビジネスインパクト）に努める必要がある

#### 現場感覚の維持

- 「象牙の塔のアーキテクト」にならない
- コードや技術の現場に留まりつつ戦略的議論にも参加
- 口で言うより行動で示す（リーダーの行動がチームの基準になる）

### コミュニケーション

#### 対立解消

- テキスト（チャット・メール）ではなく**直接対話**（ビデオ会議含む）で解決
- テキストベースは誤解を生みやすい
- 経験の浅いエンジニアの前でリーダー層の不一致を露呈しない

#### フィードバック

- ネガティブフィードバックは極力避ける（ハラスメント等の例外を除く）
- 本人が問題を認識して相談に来た際のカウンターとして行う
- AIを介したフィードバック手法も活用（壁打ち結果をウォークスルー）
- 非難なきポストモーテムは理念として正しいが、日本の組織文化では実践が難しい側面がある

### 日本と海外の文化差

- 海外（主に米国）企業文化と日本企業の違いを強く意識
- 海外向けの内容は「話半分」「読み流す」など取捨選択
- 日本では**サマライズ能力**が重視される（「蒸留された明快さ」）
- 1on1や過剰なコミュニケーションは海外文化として距離を置く
- 「スポンサー」概念は日本では明示的に認識されていないが、本質は普遍的

### 用語・表現への配慮

- 「上流・下流」という表現は好まない（依存関係で捉える）
- 翻訳用語は適宜読み替える（例：「のれん残高」→「信頼残高」）
- 自己の経験・レベルを明示する表現は避ける（傲慢に映るため）

### 読書スタイル

- 批評的に読み、日本の文脈に置き換えて解釈
- 重要な箇所は引用しページ番号を添える
- パブリックに書きにくい内容は明示的に「noteなし」とする慎重さ
- 「プラットフォームチームへの示唆」として自分の業務に読み替える視点
