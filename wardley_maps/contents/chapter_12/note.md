# 第12章 Note

## 全体所感

Wardleyが「答えを見る前に45分やれ」と言っている演習なので、正直に言えば答え（第13章）を先に読んでしまったが、それでも十分に学びがある。このシナリオは日本のSI企業やSaaS企業が直面している状況と驚くほど重なる。特にオンプレミス製品からクラウドへの移行期にある企業は、Phoenixとほぼ同じ構造的課題を抱えている。

## Phoenixの状況 — プラットフォームエンジニアの視点

SRE/プラットフォームエンジニアとしてPhoenixを見ると、以下の構造的問題が即座に目につく:

1. **センサー（コンポーネント）の産業化が進行中**: 中国製の安価なセンサーは「今はPhoenixに使えない」が、これは現行のプラクティスに最適化されているだけ。共進化の観点からは、安価なセンサーの大量配置＋新しいアルゴリズムという新しいプラクティスが出現するのは時間の問題
2. **クラウド移行がライセンスモデルのまま**: ユーティリティ課金にしない理由が「既存チャネルとの衝突を避けるため」。これは慣性そのもの
3. **エコシステム戦略の欠如**: 米国競合は公開APIでエコシステムを構築しILCモデルを実行中。Phoenix側にはAPIやエコシステムの言及すらない
4. **CIOの孤立**: 最も技術的な洞察を持つCIOが組織の隅に追いやられている。情報の非対称性の典型

## 日本企業との類似性

Phoenixのシナリオは、日本のエンタープライズソフトウェア企業が直面している状況とほぼ同型:

- **オンプレミス製品で高いシェア** → SaaS化の圧力を受けている
- **ライセンスモデルからの脱却を嫌がる** → 既存の営業チャネルとの衝突を恐れる
- **安価な代替品（OSSやクラウドネイティブツール）が台頭** → 「品質が低い」と軽視
- **海外（米国）の競合がクラウドネイティブ＋エコシステムモデルで急成長** → 「日本の商慣行は違う」と距離を置く
- **技術者の声が経営に届かない** → CIOの孤立＝エンジニアの声が意思決定に反映されない

これはまさに「情報の非対称性」問題。技術的な洞察を持つ人（CIO、SRE）と、意思決定権を持つ人（CEO、CDO、営業部長）の間に深い溝がある。

## 戦略優先順位への私見（マップを描く前の直感）

経営陣の優先順位と、SREとしての直感による優先順位の比較:

| 優先順位 | 経営陣 | 私見 |
|---|---|---|
| 1 | クラウドサービス構築 | 安いセンサー対応の新アーキテクチャR&D |
| 2 | 外部データセット活用 | クラウドサービス構築（ただしユーティリティ課金＋公開API） |
| 3 | ブラジル展開 | 外部データセット活用 |
| 4 | 既存製品開発強化 | 既存製品開発強化（短期的な顧客維持のため） |
| 5 | マーケティング | ブラジル展開は保留（市場がクラウドモデルに食われるリスク） |

最大の違いは「安いセンサー対応」の優先度。経営陣はこれを完全に軽視しているが、これこそが次のプラクティスの共進化を引き起こすトリガー。Wardleyの気候パターン「コンポーネントは共進化できる」を適用すれば、これが最も重要な戦略課題であることは明白。

## CIOの知見を活かすべき

CIOの「安いセンサーを大量に使う」という提案は、現在のPhoenixの文脈では「完全な書き直し」を意味するため拒否されているが、まさにそれが将来の方向性。CIOは技術的な直感で共進化を感じ取っているが、マップという共通言語がないためにその洞察を組織に伝えられていない。

これは信頼資本の問題でもある。CIOは組織内での影響力を失っており（役割の縮小、意思決定からの排除）、正しい洞察を持っていてもそれが意思決定に反映されない。マップがあれば、CIOの主張を視覚的に検証し、組織全体で議論できたはず。

## ポイント

- このシナリオはWardley Mapsの実践的な価値を最も直感的に示す教材。「マップなし」で戦略を議論するPhoenix経営陣の盲目性が、マップを描くことで鮮明に浮かび上がる
- 日本のエンタープライズソフトウェア企業は、このシナリオを自社の状況に置き換えて読むべき
- 技術者の洞察が経営判断に届かない「情報の非対称性」は、マップという共通言語の導入で部分的に解消できる可能性がある
- 安価なコンポーネントの登場を「品質が低い」と軽視するのは、慣性の典型的な表れ。共進化の観点から「新しいプラクティスが生まれる兆候」と読み替えるべき
