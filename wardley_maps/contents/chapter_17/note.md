# 第17章 Note

## 全体所感

この章は前2章の集大成的な位置づけで、演習を通じた実践を促す内容。正直なところ第14-17章はCanonicalの事例とハイプサイクルの話が繰り返し登場し、やや冗長に感じる。ただし、読者への3つの演習は非常に実践的であり、自チームに適用する価値がある。

## 演習1の適用：プラットフォームチームのドクトリン評価

Wardleyのドクトリンフェーズ表（Figure 236参照 -- Mediumで閲覧可能）を自チームに適用すると、以下の傾向が見える：

- Phase I（自傷行為の停止）：ユーザーニーズの理解、重複排除は概ね達成
- Phase II（コンテキスト認識）：適切な手法の使い分け、FIREは部分的に実践
- Phase III（Better for Less）：継続的改善、フロー最適化は進行中
- Phase IV（継続的進化）：エコシステムへの傾聴、戦略的プレイはまだ不十分

多くの日本企業のプラットフォームチームはPhase II～IIIの境界にいるのではないか。Phase IVの「エコシステムへの傾聴」は、Internal Developer Platformの文脈では「開発者コミュニティの声を組織横断で吸い上げる仕組み」に相当する。

## 演習2の適用：プラットフォームサービスの10年後

プラットフォームチームの主要サービスを10年後に投影すると：
- CI/CD：GitHub Actionsレベルのサービスがさらにコモディティ化し、AI支援でパイプライン自体が自動生成される
- モニタリング：オブザーバビリティがAIによるAIOpsに進化し、異常検知→原因分析→修復提案が自動化
- IaC：Terraformレベルのツールがさらに抽象化され、意図ベース（Intent-based）のインフラ管理に

フィードバックループの例：AI支援ツールの導入→開発者生産性向上→より多くのサービス生成→モニタリング負荷増大→プラットフォームチームの負荷増大→AI支援の必要性増大、というループが予想される。

## 競合バイアスの読み替え

Wardleyが指摘するカスケードバイアス（皆が言っているから正しい）は、プラットフォーム技術の選定でも頻出。「Kubernetesを使うべき」「マイクロサービスにすべき」というカスケードバイアスに流されず、自社のマップ上でのポジションに基づいて判断すべき。

## ポイント

- ドクトリンフェーズ表を使って自チームの現在地を客観的に評価し、次のフェーズに向けた計画を立てる
- 10年後マッピングではフィードバックループに特に注意。AI導入による生産性向上が別の負荷を生む二次効果を予測する
- カスケードバイアスに注意。技術選定は「みんなが使っているから」ではなく、自社の進化段階に基づいて行う
