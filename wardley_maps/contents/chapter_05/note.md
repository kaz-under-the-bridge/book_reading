# 第5章 Note

## 全体所感

この章はWardley本の中で最も実務的かつ痛切な章。ILCモデルの理論的な美しさと、Zimkiの実践的な失敗の対比が鮮やかで、戦略の実行は「正しさ」だけでは足りないことを生々しく描いている。SRE/プラットフォームエンジニアとして、技術的に正しい方向を示しても組織が動かない経験は何度もあり、この章の教訓は身に染みる。

## ILCモデルとプラットフォームチームの価値

ILCモデル（Innovate-Leverage-Commoditise）は、プラットフォームチームの存在意義を理論的に説明する最良のフレームワークの一つ。

- プラットフォームチームは本質的にTown Plannerとして「Commoditise」の役割を担う
- その上で開発チーム（外部のPioneer）がイノベーションを行う
- メタデータの活用（API利用パターンの分析）は、プラットフォームチームが提供すべき次のサービスを特定する手段
- これは「コストセンター」としてのプラットフォームチームの価値を「未来のイノベーションを可能にするエンジン」として再定義する

AWS自体がILCモデルの最大の成功例。S3、EC2、Lambdaと段階的にCommoditiseしながら、ユーザーのイノベーションをLeverageし、新サービスを生み出し続けている。

## 政治的資本と信頼資本

著者の失敗 -- 取締役会のニーズを無視 -- は「信頼資本」の概念に直結する。

- 著者は技術的なビジョンに没頭するあまり、取締役会という重要なステークホルダーへの「信頼資本」の蓄積を怠った
- 日本企業でも同じ構図がある。技術リーダーが「これが正しい方向だ」と主張しても、経営層の信頼を得ていなければ予算もリソースも動かない
- 特にプラットフォームチームは、直接的な収益を生まないため、経営層との信頼関係が命綱

対策として:
- 決算説明会資料や中期経営計画を読み、経営層の投資方針を把握する（第3章のZimki事例ではこれが欠落していた）
- マップを「技術的に正しい」ツールとしてだけでなく、「経営判断のための共通言語」として経営層に提示する
- 段階的に信頼を構築し、大きな変革は十分な信頼資本を蓄積してから実行する

## Acceleratorとしてのオープンソース

オープンソースによるコモディティ化の加速は、プラットフォームチームの戦略的ツール。

- KubernetesがBorg（Google社内ツール）のオープンソース版として登場し、コンテナオーケストレーションをコモディティ化した。これはまさにAcceleratorの典型例
- プラットフォームチームとして、社内ツールをオープンソースにするかどうかは戦略的判断。コモディティ化を加速するなら公開、差別化を維持するなら非公開
- 日本企業では社内ツールのオープンソース化に対する心理的障壁が高いが、ILCモデルの文脈で考えれば合理的な選択になりうる

## 「Pre-shaved Yaks」と目的の設定

「Pre-shaved Yaks」という目的設定は天才的。開発者なら誰でも共感できる「無駄な作業をなくす」という目的は、チームの士気を高め、方向性を統一する。

- プラットフォームチームの目的も「開発者の認知負荷を下げる」「コードを書くこと以外の時間を最小化する」のように、ユーザー（開発チーム）視点で定義すべき
- 「インフラの効率化」「コスト最適化」は組織視点の目的であり、チームを動機づけるにはユーザー視点が必要

## ポイント

ILCモデルはプラットフォームチームの存在意義を理論的に裏付ける強力なフレームワーク。しかし著者自身の失敗が示すように、戦略の正しさと実行の成功は別問題。技術的なビジョンと政治的な信頼資本の両方が揃って初めて変革は実現する。日本企業でプラットフォーム戦略を推進する際には、この教訓を常に念頭に置くべき。
