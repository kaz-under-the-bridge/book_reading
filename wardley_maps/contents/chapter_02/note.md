# 第2章 Note

## 全体所感

マップの構造がようやく具体的になった章。Y軸がバリューチェーン（可視性）、X軸が進化（Genesis→Commodity）という2軸は直感的に理解しやすい。特にプラットフォームエンジニアリングの文脈では、自チームが提供するサービスのどこにユーザー（開発者）がいて、どのコンポーネントがどの段階にあるかを可視化するのに使える。

## PFEチームへのマップ適用

試しにプラットフォームチームのマップを脳内で描いてみると:

- **ユーザー**: 開発者チーム
- **Visible（上位）**: デプロイパイプライン、開発者ポータル、セルフサービスインフラ
- **Invisible（下位）**: Kubernetes、ネットワーク、IAM、モニタリング基盤
- **Genesis**: AI支援デプロイ、LLMベースの障害分析
- **Commodity**: DNS、CDN、オブジェクトストレージ、CI/CDの基本パターン

「うちのCI/CDは特別」と思っていても、業界的にはProduct〜Commodityに位置するものが多い。ここに投資し続けるのは「自家製CRMを作る」のと同じかもしれない。逆にAI支援の領域はまだGenesisで、ここが差別化ポイントになりうる。

## 「マップによる議論」の価値

著者が繰り返し強調する「マップの価値は正確さではなく議論を引き起こすこと」は、PdM本の「ベストプラクティスのワーストなところ」と呼応する。ロードマップやアーキテクチャ図はしばしば「見せるための資料」になりがちだが、Wardley Mapは「前提を暴露するための道具」として設計されている点が本質的に異なる。

## 進化の4段階とSRE実務

進化段階の概念はSRE実務にも直結する:

| 段階 | SRE的な対応 |
|------|-----------|
| Genesis | 実験的サービス。SLOなし、手動運用OK |
| Custom Built | チーム固有のツール。属人的な運用 |
| Product | 標準化されたサービス。SLO設定、ランブック整備 |
| Commodity | フルマネージドサービス利用。運用は最小限 |

サービスの進化段階に応じて運用のアプローチを変えるべき、という示唆はSRE本のトイル削減の議論と整合する。

## ポイント

- Wardley Mapはプラットフォームチームの技術投資判断ツールとして具体的に使えそう
- 「議論を引き起こす道具」としてのマップは、チーム間の認識ギャップ（情報の非対称性）解消にも有効
- 進化段階の概念は、SRE実務における運用レベルの設計基準になりうる
