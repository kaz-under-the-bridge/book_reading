# 第11章 Note

## 全体所感

この章はPDFで100ページ以上に及ぶ長大な章であり、Wardley Maps理論の「教科書」として機能する。気候パターン、ドクトリン、ゲームプレイの3層構造は、プラットフォームチームの技術戦略を考える際のフレームワークとして極めて有用。特にドクトリンの一覧は、チームの健全性チェックリストとしてそのまま使える。

## Wardley's Doctrineをプラットフォームチームの言葉に翻訳する

Wardley's Doctrineの多くは、プラットフォームエンジニアリングの原則とほぼ等価:

| Wardley's Doctrine | プラットフォームチームでの実践 |
|---|---|
| Be transparent | 内部開発者ポータルでのサービスカタログ公開 |
| Use a common language | マップを使った技術戦略の共有 |
| Remove bias and duplication | サービスカタログによる重複検出と統合 |
| Use appropriate methods | Genesis領域はAgile、Commodity領域はSix Sigma |
| Focus on user needs | 開発者体験（DX）の継続的改善 |
| FIRE | 小さなコンポーネント、再利用可能なモジュール |
| Manage failure | Chaos Engineering、ゲームデー |
| Think small | マイクロサービス、小さなチーム |
| Design for constant evolution | Pioneer-Settler-Town Plannerモデル |

## レジリエンスの4類型 — SREの文脈

Wardleyの4類型（fragile/robust/fluid/resilient）はSREの概念と直結する:

- **Fragile**: モノリシック、単一障害点、手動運用。典型的なレガシーシステム
- **Robust**: 冗長化、自動フェイルオーバー、定義済みのDRプラン。伝統的なエンタープライズITの目標
- **Fluid**: 小さなコンポーネントが独立して進化。マイクロサービスアーキテクチャだが、個々の品質は低い可能性
- **Resilient**: 冗長化されたマイクロサービスが進化を続ける。Chaos Engineeringで常に鍛えられている。Netflix的なアーキテクチャ

多くのSREチームはrobustを目指しているが、Wardleyの主張はfluid→resilientの方向性。つまり「壊れない」ことより「壊れても適応する」ことを優先すべき。これは日本企業の品質文化（「壊れないものを作る」）とは相容れない面があるが、クラウドネイティブの時代では避けられない方向転換。

## 「Focus on core!」の罠

Wardleyの「コアは一過性」という主張は、プラットフォームチームの存在意義に関わる。プラットフォームチームはしばしば「コアではない」と見なされ、投資が削減される。しかしWardleyの論理に従えば、何がコアかは常に変化しており、今のコア（プロダクト開発能力）が将来もコアである保証はない。

むしろプラットフォームチームの本質的な価値は、「コアが変化した時に迅速に適応できる基盤を提供すること」 — すなわちWardleyが言うresilienceの提供にある。この説明は経営層へのピッチとして使える。

## Sensing Engines (ILC) とプラットフォームの成長戦略

ILCモデルは内部プラットフォームの成長戦略そのもの:

1. **Industrialise（産業化）**: 社内の共通コンポーネントを特定し、セルフサービスプラットフォームで提供
2. **Leverage（レバレッジ）**: 開発チーム（エコシステム）がその上に新しいサービスを構築
3. **Commoditise（コモディティ化）**: 消費パターンのデータから次に産業化すべきコンポーネントを特定

このサイクルを回すことで、プラットフォームチームは「コストセンター」から「イノベーション基盤の提供者」にポジションを変えられる。

## 気候パターン — 「ほとんどの競合は状況認識が低い」

これは逆説的に朗報。競合他社が盲目であるなら、自組織が少しでもマッピングを始めるだけで優位に立てる。日本のIT業界では、Wardley Mapsの認知度はまだ低い。つまり、この時点でマッピングを始めること自体が差別化要因になりうる。

## ポイント

- ドクトリンは「全組織に適用すべき普遍的原則」、ゲームプレイは「マップ上の位置に依存するコンテキスト固有の戦略」。この区別を理解せずにベストプラクティスを適用すると失敗する
- レジリエンスの4類型はSREのアーキテクチャ判断に直接使える。Robust（壊れない）からResilient（壊れても適応する）への移行を明確に目標設定する
- ILCモデルはプラットフォームチームの成長戦略。消費データからの逆算で次の投資を決定する仕組みを作る
- 「コアは一過性」という認識は、プラットフォームチームの長期的な存在意義を説明する鍵。適応能力こそが永続的な価値
